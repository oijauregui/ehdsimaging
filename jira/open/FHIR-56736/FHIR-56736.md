# FHIR-56736: Observation.basedOn accession.identifier is 1..1 but ServiceRequestOrder doesn't require accession; align both at 0..*

## Description

Observation.basedOn:ServiceRequestOrderEuImagingaccession.identifier has a cardinality of 1..1, but the ServiceRequestOrder doesn't require accession number.

Proposed resolution:

Make these two elements consistent with cardinality 0..*. Accession number should not be required to account for legacy data, and it should not be limited to 1 because a single study can satisfy multiple orders or requests for imaging, in which case there could be multiple accession numbers in a single report.

Reporter(s): Patrick Gorman, Epic
